# tailwind-hue-theme — Technical Guide

A deep-dive into what this plugin is, why it exists, who it's built for, and how it works under the hood.

---

## What It Is

**tailwind-hue-theme** is a Tailwind CSS v4 plugin that remaps your color palette to perceptually-uniform OKLCH values, all controlled by a single CSS custom property: `--brand-h`.

Instead of hard-coding a brand color and regenerating a full palette every time a client changes their mind, you change one number — a hue angle between 0 and 360 — and your entire UI updates: backgrounds, borders, text, accents, and everything in between.

The plugin ships two things:

1. **The plugin itself** — injects CSS tokens into `:root` that map the Tailwind `slate`, `indigo`, and `cyan` scales to OKLCH color expressions.
2. **The HuePicker widget** — an optional, zero-dependency floating UI for interactively selecting hues at runtime. Useful in prototypes, demos, white-label tools, or any product where end users set their own theme.

---

## Why I Built This

Tailwind has excellent color semantics, but its palette is static. You pick your colors at build time and ship them. If a client needs a rebrand — or if you're building a tool where every user gets their own color — you're either regenerating a CSS file, maintaining parallel Tailwind configs, or reaching for a CSS-in-JS solution that adds complexity and runtime overhead.

The deeper problem is that it's not just one color. A brand palette needs a full range of tints and shades: a background tone, borders, text, interactive states, and accents. Getting those to look *consistent* across hues is where HSL falls apart. Two colors at the same HSL saturation and lightness can look dramatically different in perceived brightness and vividness depending on the hue — yellow reads much lighter than blue at the same `hsl` values.

**OKLCH solves this.** It's a perceptually uniform color space where chroma (colorfulness) and lightness are predictable regardless of hue. The same chroma value at hue 250 (indigo) and hue 155 (emerald) will look equally colorful to the human eye. That's the core insight this plugin is built on.

The result: one CSS variable, one hue angle, and you get a stable, visually-coherent palette at any point in the spectrum — without JavaScript, without build steps, without re-running Tailwind.

## Target Audience

This plugin is for developers who:

- **Build multi-tenant or white-label products** where each customer picks their own brand color. One deployed stylesheet serves every brand.
- **Work on design systems or component libraries** and want to validate that components look correct at multiple hues without maintaining separate themes.
- **Build demos or prototypes** and want a color picker that actually respects visual consistency — not just a raw hue slider that happens to set `--primary`.
- **Use Tailwind v4** and want semantic color tokens that respond to CSS variables at runtime without JavaScript frameworks or CSS-in-JS.
- **Care about color correctness** — designers who've dealt with "why does the yellow look so washed out" know the pain this solves.

If you're building a single product with a fixed brand color and no need for runtime palette switching, this plugin adds capability you don't need. Use Tailwind's built-in color tokens instead.

## The OKLCH Advantage

Traditional CSS color functions like `hsl()` are convenient but perceptually inconsistent. The `L` in HSL stands for "lightness," but it doesn't correspond to how light or dark colors actually *appear*. A yellow at `hsl(60 100% 50%)` looks blazingly bright; a blue at `hsl(240 100% 50%)` looks comparatively dim — same "lightness", very different perception.

**OKLCH** (Oklab Lightness Chroma Hue) fixes this:

| Parameter | Meaning | Range |
|-----------|---------|-------|
| `L` | Perceived lightness | `0` (black) → `1` (white) |
| `C` | Chroma (colorfulness) | `0` (gray) → ~`0.4` (maximally vivid) |
| `H` | Hue angle | `0`–`360` |

Because `L` in OKLCH is perceptually calibrated, you can pin lightness and chroma and sweep `H` across the full spectrum — every resulting color will have the same perceived brightness and saturation. That's what makes the single-variable approach work correctly.

The plugin uses fixed `L` and `C` values for each step on each scale, and sets `H` to `var(--brand-h, <fallback>)`. The fallback means tokens degrade gracefully if the variable isn't set.

```
tailwind-hue-theme/
├── src/
│   ├── index.ts        ← Plugin entry: CSS token generator
│   └── widget.ts       ← Optional HuePicker DOM widget
├── dist/               ← CJS + ESM bundles, .d.ts files (generated by tsup)
├── tests/
│   ├── plugin.test.ts  ← Tests for buildTokens and createHuePlugin
│   └── widget.test.ts  ← Tests for HuePicker lifecycle and DOM
└── docs/               ← Astro Starlight documentation site
```

Two named package exports are declared in `package.json`:

```json
{
  "exports": {
    ".":       { "import": "./dist/index.mjs", "require": "./dist/index.js" },
    "./widget": { "import": "./dist/widget.mjs", "require": "./dist/widget.js" }
  }
}
```

The plugin and the widget are separate entry points so the widget (which depends on DOM APIs) is never bundled unless explicitly imported. Server-side rendering environments import only `"tailwind-hue-theme"` and are never exposed to `document` references.

## How the Plugin Works

The plugin calls Tailwind's `addBase()` API to inject 33 CSS custom properties into `:root`. Every property is an `oklch()` expression:

```css
:root {
  --brand-h: 250;

  /* slate scale (low chroma — stays visually dark) */
  --color-slate-50:  oklch(0.984 0.004 var(--brand-h, 250));
  --color-slate-950: oklch(0.13  0.028 var(--brand-h, 250));

  /* indigo scale (full chroma — follows brand hue directly) */
  --color-indigo-50:  oklch(0.962 0.03  var(--brand-h, 250));
  --color-indigo-500: oklch(0.585 0.18  var(--brand-h, 250));

  /* cyan scale (offset hue for a complementary secondary) */
  --color-cyan-50:  oklch(0.962 0.03  calc(var(--brand-h, 250) + 40));
  --color-cyan-500: oklch(0.585 0.18  calc(var(--brand-h, 250) + 40));
}
```

### Color scale design decisions

**Slate** uses chroma values between `0.004` and `0.036`. At these levels, the gray scale picks up just enough of the brand hue to feel cohesive — dark backgrounds at `slate-900`/`slate-950` remain visually dark regardless of hue. They won't turn emerald-green or rose-pink; they'll just have a matching *temperature*.

**Indigo** uses chroma between `0.03` and `0.18` — the full expressive range. This is your primary interactive color: buttons, links, focus rings, highlights.

**Cyan** mirrors indigo's chroma range but offsets its hue by `secondaryOffset` degrees (default: `+40`). At hue 250 (indigo), cyan becomes a complementary teal-blue. At hue 10 (rose), cyan becomes a warm yellow-orange. The secondary always complements rather than clashes because the offset is constant.

### `buildTokens(options?)`

This utility function returns the raw CSS token map as a plain object without registering a Tailwind plugin. Use it in tests or build scripts:

```ts
import { buildTokens } from 'tailwind-hue-theme'

const tokens = buildTokens({ defaultHue: 155 })
// tokens['--color-indigo-500'] → 'oklch(0.585 0.18 var(--brand-h, 155))'
```

### `createHuePlugin(options?)`

Returns a Tailwind plugin instance. Accepts:

| Option | Type | Default | Description |
|---|---|---|---|
| `defaultHue` | `number` | `250` | Hue written to `--brand-h` in `:root` |
| `secondaryOffset` | `number` | `40` | Degrees added to `--brand-h` for the cyan scale |

The default export is `createHuePlugin()` with all defaults applied, which is what gets invoked when you write `@plugin "tailwind-hue-theme"` in CSS.

## How the HuePicker Widget Works

The widget is a fully self-contained DOM artifact — no dependencies, no framework, no build step needed to use it. It renders:

- A **trigger button** (fixed, bottom-right) with a rainbow conic-gradient ring and a center dot that tracks `--brand-h` in real time.
- A **popup panel** with a circular color wheel (draggable), preset swatches, and a reset button.

The widget injects its own `<style>` tag once on first instantiation (guarded by `document.getElementById('hue-picker-styles')` to prevent duplicates).

### Color wheel interaction

Wheel drag is implemented using pointer events and `setPointerCapture` for reliable tracking even when the cursor leaves the circle. The hue angle is computed via:

```ts
const angle = Math.atan2(dy, dx) * (180 / Math.PI) + 90
```

The `+90` offset rotates the coordinate system so hue 0 (red) sits at 12 o'clock, matching user intuition from standard color wheels.

The drag handle is placed at 82% of the wheel's radius using basic trigonometry, keeping it clearly inside the ring without obscuring the color gradient.

### Angle normalisation

All hue values are normalised before being applied:

```ts
((hue % 360) + 360) % 360
```

This ensures values like `-10` correctly resolve to `350` (red-pink) and values like `400` resolve to `40` (yellow-orange), regardless of how `setHue()` is called.

### Persistence

The selected hue is saved to `localStorage` under the key `storageKey` (default: `"tailwind-hue-theme"`). On construction, the widget reads this key and restores the previous selection. Persistence can be overridden per-instance with a different `storageKey`, or suppressed entirely by always providing an explicit `defaultHue` and not using the built-in widget UI.

### Accessibility

The widget is designed to be keyboard-usable:

- `role="dialog"` on the panel, with `aria-modal="true"` and `aria-label`
- `role="slider"` on the color wheel, with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, `aria-label`
- Arrow keys adjust hue by ±5° per step
- `Escape` closes the panel
- `aria-expanded` on the trigger reflects open/closed state

## Integration Patterns

### Tailwind v4 CSS-first (recommended)

```css
@import "tailwindcss";
@plugin "tailwind-hue-theme";
```

Add the HuePicker for interactive switching:

```html
<script type="module">
  import { HuePicker } from 'tailwind-hue-theme/widget'
  new HuePicker()
</script>
```

### Tailwind v3 / v4 JS config

```js
// tailwind.config.js
import { createHuePlugin } from 'tailwind-hue-theme'

export default {
  plugins: [createHuePlugin({ defaultHue: 155 })],
}
```

### Custom default hue (no JS)

Set the default hue directly in CSS after loading the plugin:

```css
@import "tailwindcss";
@plugin "tailwind-hue-theme";

:root {
  --brand-h: 155; /* Emerald green */
}
```

### White-label multi-tenant

Load `--brand-h` from your server-rendered page — inject it per-user in HTML:

```html
<html>
<head>
  <style>:root { --brand-h: <%= user.brandHue %>; }</style>
```

No JavaScript, no flash of wrong color, no extra network request.

### User-selectable themes

Pair the plugin with the HuePicker for full end-user control:

```ts
import { HuePicker, DEFAULT_PRESETS } from 'tailwind-hue-theme/widget'

const picker = new HuePicker({
  defaultHue: 250,
  presets: [
    ...DEFAULT_PRESETS,
    { label: 'Brand Blue', hue: 220 },
  ],
  onChange: (hue) => {
    // Sync to user preferences API
    fetch('/api/preferences', {
      method: 'POST',
      body: JSON.stringify({ brandHue: hue }),
    })
  },
})
```

### Driving the palette from a native range input

```ts
import { HuePicker } from 'tailwind-hue-theme/widget'

const picker = new HuePicker({ presets: [] })

document.getElementById('hue-slider').addEventListener('input', (e) => {
  picker.setHue(Number(e.target.value))
})
```

## TypeScript

The package is fully typed. Import the types alongside the functions:

```ts
import {
  createHuePlugin,
  buildTokens,
  type HuePluginOptions,
} from 'tailwind-hue-theme'

import {
  HuePicker,
  DEFAULT_PRESETS,
  getWidgetCSS,
  type HuePickerOptions,
  type HuePreset,
} from 'tailwind-hue-theme/widget'
```

`HuePluginOptions`, `HuePickerOptions`, and `HuePreset` are all exported interfaces. `getWidgetCSS()` returns the raw CSS string injected by the widget — useful for SSR environments that pre-inject styles rather than letting the widget inject its own `<style>` tag.

## Build and Distribution

The package is built with **tsup**, which produces:

| File | Format | Purpose |
|------|--------|---------|
| `dist/index.js` | CJS | Node.js / Webpack / legacy bundlers |
| `dist/index.mjs` | ESM | Vite / Rollup / modern bundlers |
| `dist/index.d.ts` | TypeScript declarations | IDE autocompletion and type checking |
| `dist/widget.js` | CJS | Widget — server-safe import |
| `dist/widget.mjs` | ESM | Widget — modern bundlers |
| `dist/widget.d.ts` | TypeScript declarations | Widget types |

`tailwindcss` is declared as a peer dependency and externalized from the bundle. Consumers bring their own Tailwind installation; the plugin only adds tokens on top of it.

Tests run in **vitest** with the `jsdom` environment, covering token generation correctness (all 33 properties, hue math, secondary offset) and the full HuePicker widget lifecycle (mount, drag, persistence, destroy, accessibility attributes).
